# FufuLauncher Go + Win32 重构与实施路线

状态：开工前设计基线  
制定日期：2026-07-17  
上游仓库：<https://github.com/FufuLauncher/FufuLauncher>  
上游基线：`master@b5a050ebd319341bddc4189491c90c22162d33fa`  
适用平台：Windows 10/11 x64

## 1. 项目定义

本项目用 Go + Win32 重新实现 FufuLauncher 中仍在范围内的用户行为，不追求源码结构、技术栈或界面像素级一致。上游是需求参考和更新信号源，不是运行时依赖。

“实现所有功能”在本项目中的准确含义是：

1. 实现 `docs/upstream-scope-matrix.md` 中标为“保留”的功能。
2. 不实现用户明确排除的模块，也不保留它们的入口、后台服务、数据库表、网络请求或隐式依赖。
3. 上游新增功能必须先经过范围规则分类；不能因为自动同步而绕过删减边界。
4. UI 只要求操作能力和状态反馈完整，不复刻上游复杂 WinUI、动态背景、内容卡片和动画。

## 2. 不可变边界

### 2.1 明确排除

- FufuLauncher 自身的登录、用户协议登录链路及安全认证页面。
- 米游社、HoYoLAB、云游戏和社区账号登录、Cookie、二维码、验证码、Token 刷新。
- 游戏账号保存、登录账号列表和游戏设置页内的账号切换。
- 签到、批量签到、云游戏签到、便笺、树脂/委托/洞天等账号数据。
- 抽卡记录、抽卡统计、UIGF、成就、背包、玩家角色、旅行者札记。
- 新闻、社区资讯、BBS、VanillaBBS、公告内容浏览器和首页资讯卡。
- 原仓库“控制面板/工具箱”中的全部功能和入口，不拆分保留其中的子项。
- 数据中心、帮助文档、养成计算器、提瓦特地图内容页。
- “其他/实用工具”中的附加程序随游戏启动。
- “其他/实用工具”中的内置浏览器。
- WebView2 及仅为上述排除功能服务的缓存、登录和网络组件。

### 2.2 保留原则

- 本地游戏发现、路径选择、版本/区服识别。
- 游戏启动、启动参数、显示器/分辨率/窗口模式、启动后行为。
- 游戏资源下载、预下载、校验与修复。
- 官服/B 服切换和高级服务器转换，但必须采用可回滚文件事务。
- 桌面快捷方式。
- 注入入口、模块选择、插件管理、插件配置和插件更新；它们是高风险独立子系统，不能拖垮主程序。
- 游戏内截图、截图快捷键和保存目录。
- FPS/CPU/GPU 覆盖层及相关开关。
- HDR 开关及亮度配置。
- 鼠标左/右键连点、键盘连按、触发键、停止键和配置持久化。
- 主题、语言、托盘、窗口尺寸、单实例、日志、更新和诊断。
- 启动声音以及 BetterGI 联动（均须在实现前完成依赖和调用行为审计）。

完整映射见 [功能范围矩阵](upstream-scope-matrix.md)。

## 3. 产品形态

### 3.1 主界面

采用单主窗口、左侧窄导航、右侧内容区：

1. 首页：游戏状态、路径、版本、区服、启动按钮、常用启动配置摘要。
2. 游戏管理：路径、下载/预下载、校验修复、区服切换、快捷方式。
3. 启动设置：分辨率、显示器、窗口模式、自定义参数、HDR、启动后行为。
4. 输入增强：键盘连按、鼠标连点、触发/停止键、间隔、实时状态。
5. 注入与插件：注入开关、模块、插件库、插件设置和更新状态。
6. 截图与覆盖层：截图热键/目录、FPS 覆盖层。
7. 程序设置：暗色主题、语言、托盘、窗口、更新、日志和诊断。

### 3.2 视觉规则

- 默认深灰背景、单一强调色、8px 左右圆角、清楚的边框和层级。
- 使用 Win32 原生窗口、按钮、编辑框、列表、进度条；必要处做 owner-draw/custom-draw。
- 使用 DWM 深色标题栏；不使用 WebView2、CEF、WinUI 3、WPF、Electron 或视频背景。
- 动画仅限 120–180ms 的颜色/透明度过渡；低配机器或远程桌面可完全关闭。
- 所有功能必须可由键盘操作；焦点框、禁用状态和错误状态不可只靠颜色表达。
- DPI 以每显示器 V2 感知；字体和布局按 DPI 重算，不靠整体位图缩放。

## 4. 技术架构

建议目录在进入实现阶段后按以下职责建立：

```text
cmd/genshintools/            程序入口、启动参数、进程级恢复
internal/app/                生命周期、用例编排、状态机
internal/ui/                 页面模型、UI 消息、暗色主题
internal/platform/win32/     HWND、消息循环、DWM、COM、托盘、DPI
internal/input/              全局钩子、SendInput、连点/连按状态机
internal/game/               路径发现、配置、进程检测、启动
internal/resources/          清单、下载、预下载、校验、修复
internal/server/             官服/B服识别、切换、回滚
internal/injection/          注入适配接口和外部模块进程边界
internal/plugins/            插件发现、校验、配置、更新
internal/capture/            截图和快捷键
internal/overlay/            FPS/CPU/GPU 覆盖层
internal/config/             EXE-local data、迁移、原子写入
internal/update/             本程序更新、校验、回滚
internal/upstream/           上游差异解析和范围分类
internal/diagnostics/        结构化日志、崩溃标记、句柄/GDI统计
scripts/                     构建、打包、上游检查
docs/                        设计、范围、测试和发布文档
```

### 4.1 依赖原则

- Win32 调用优先集中在 `internal/platform/win32`，业务层不直接散落 syscall。
- 基础调用使用 `golang.org/x/sys/windows`；缺失 API 用小型、带类型和测试的封装补齐。
- UI 不选择大型跨平台框架，避免引入额外渲染线程、GPU 依赖和大体积运行时。
- 核心模型不依赖 HWND，确保路径、参数、下载、校验和状态机可做普通 Go 单元测试。
- 注入二进制不链接进主进程；通过版本化适配层调用并在启动前验证架构、版本和哈希。
- 配置、缓存、日志、下载暂存统一放在 EXE 同级 `data/` 下；程序目录只保留可执行文件和只读资源。

### 4.2 许可证和来源

- 上游仓库采用 MIT License；重构时保留必要版权与许可证文本，并记录参考过的上游提交。
- 不把“重新用 Go 实现”误当成可以忽略第三方许可证。任何 DLL、插件、字体、图标、声音和协议定义都要单独核对来源与再分发权。
- 发布包生成 `THIRD_PARTY_NOTICES.md` 和依赖/二进制 SHA-256 清单；来源或许可证不清楚的资源不进入包。

### 4.3 线程模型

- 主 goroutine 锁定 OS 线程，独占 Win32 UI 消息循环。
- UI 只处理绘制、轻量状态变化和派发，不执行网络、哈希、递归扫描、进程等待或磁盘迁移。
- 后台任务通过带任务 ID 的 `PostMessage`/调度队列回传；旧任务结果不能覆盖新任务状态。
- 输入钩子拥有单独的锁定 OS 线程和消息循环，不与 UI 或下载线程共用。
- COM 组件由创建它的线程初始化、使用和释放，不跨 apartment 传裸接口指针。
- 退出走单一 shutdown coordinator，按“停止接单→取消任务→释放钩子/计时器→销毁窗口→退出消息循环”执行。

## 5. 连点与连按专项设计（P0）

上游当前实现采用 `WH_KEYBOARD_LL`/`WH_MOUSE_LL`、`SendInput` 和固定 50ms 周期。重构不能只做等价翻译；输入能力要先做成可独立验证的引擎。

### 5.1 支持能力

- 键盘模式：按住触发键时，以扫描码重复目标键的完整 down/up；松开立即停止。
- 鼠标模式：按住左键或右键时，重复对应按钮的完整 down/up；松开立即停止。
- 独立停止热键：无论当前焦点和模式，按下后立即解除武装并停止输出。
- 间隔可配置，初始默认 50ms；给出安全下限，防止 0ms 忙循环。
- 显示“禁用/待触发/运行/停止中/故障”状态和最近一次错误。
- 设置持久化，但程序重启后只恢复“已配置”，默认不自动进入输出状态。

### 5.2 状态机

```text
Disabled -> Armed -> Running -> Armed
    ^          |        |
    +----------+--------+  stop / disable / shutdown / fault
```

- 状态转移由一个输入引擎串行拥有；UI、钩子和停止命令不能各自改多个布尔值。
- 每次启动生成递增 generation；旧计时器、旧回调和旧 goroutine 发现 generation 不匹配立即退出。
- 启动和停止必须幂等；连续点击开关、键盘自动重复、鼠标抖动不能创建第二个输出循环。

### 5.3 Win32 实现规则

- hook 回调只读取结构、过滤事件、写入无阻塞事件槽并立即调用 `CallNextHookEx`。
- 使用自定义 `dwExtraInfo` 标记本程序注入事件，同时检查 LL hook 的 injected 标志，杜绝自激回路。
- 键盘输出使用扫描码；正确处理扩展键、左右修饰键、NumPad 和布局映射。
- 每次输入必须成对发送；停止、panic、退出和配置切换时做防御性 key-up/mouse-up 清理。
- 检查 `SendInput` 返回数量和 `GetLastError`，失败时停止循环并向 UI 报告，不能静默空转。
- 高精度节拍使用 waitable timer；不在 hook 回调中 `Sleep`、等待 channel 或写日志。
- 检测完整性级别。若游戏以更高权限运行，明确提示需以相同权限重启，避免“界面显示运行但游戏收不到输入”。
- 不吞掉用户真实按键/鼠标事件；停止热键默认仍传给下一个 hook，除非以后有明确的可选设置。

### 5.4 必过验收

- 捕获钩子中验证键盘、左右键真实 `SendInput` 的 down/up 数量一致，短节拍矩阵无粘键；长跑作为 S13 可选发布回归。
- 1000 次快速按下/松开触发键；任意时刻最多一个输出循环。
- 连续切换启用/禁用 200 次，goroutine、线程、句柄数回到稳定基线。
- 运行中按停止键、关闭窗口、退出托盘、切换模式、休眠/唤醒，均在限定时间内停止。
- 普通权限/管理员权限组合均给出正确结果或明确错误；不能假装成功。
- 30/50/100/250ms 间隔进行测量，抖动和实际频率记录到测试报告。
- 与截图热键同时启用，互不自触发、互不抢占、退出时都正确卸载 hook。

输入专项未通过时，不进入注入、插件和正式发布阶段。

## 6. 分阶段路线

本节保留架构层面的阶段说明；实际代码工作的唯一编号和先后顺序以 [需求总表与唯一代码执行顺序](implementation-order.md) 为准。

### Phase 0：范围与基线（本阶段）

交付：

- 本路线文档、范围矩阵、稳定性清单、上游同步方案。
- `upstream.lock.json` 锁定上游仓库、分支、提交和范围策略版本。

退出条件：用户确认保留/排除矩阵；文档中不存在“所有其他功能”这类不可测试表述。

### Phase 1：可生存的 Win32 壳

交付：

- 单实例、主窗口、暗色标题栏、DPI、导航、托盘、EXE-local `data/`。
- 结构化日志、panic 恢复、上次异常退出标记、诊断页。
- 可取消后台任务框架和统一 shutdown coordinator。
- 调试构建保留控制台/错误文本，GUI 发布构建不隐藏诊断文件。

退出条件：窗口 1000 次创建/销毁测试、最小化/恢复/托盘压力测试通过；稳定性清单 A–D 类关闭。

### Phase 2：输入引擎 P0

交付：

- 键盘连按、鼠标左/右连点、触发键、停止键、间隔、状态和持久化。
- 独立输入测试工具或测试页、事件计数、延迟统计和 soak test。

退出条件：第 5.4 节全部通过；任何“偶尔失效、松开仍连、关闭后仍输出”均为发布阻断。

### Phase 3：本地游戏发现与启动

交付：

- 自动探测和手动选择路径、识别多个候选 EXE、读取 `config.ini`。
- 游戏运行状态、普通启动、自定义参数、分辨率、显示器、窗口模式、启动后行为。
- HDR 注册表设置、快捷方式、进程监控。
- 启动声音，以及审计通过后的 BetterGI 联动。

退出条件：路径不存在、多 EXE、Unicode/长路径、权限不足、进程快速退出均有确定反馈；UI 不冻结。

### Phase 4：资源管理与区服切换

交付：

- 游戏/语音包下载、预下载、断点/重试、并发限制、进度、取消。
- 清单校验、文件校验与修复、磁盘空间预检、staging 和原子落盘。
- 官服/B服识别、切换、备份、失败回滚。

退出条件：断网、磁盘满、哈希错误、取消、程序崩溃后能恢复或清理；不破坏已有游戏文件。

### Phase 5：截图与覆盖层

交付：

- 游戏窗口截图、组合热键、保存目录和失败提示。
- FPS/CPU/GPU 覆盖层、位置/显示开关、目标进程退出后的清理。

退出条件：截图与输入 hook 并存；DPI、多显示器、全屏/无边框和游戏退出场景通过。

### Phase 6：注入与插件（高风险）

交付：

- 注入适配接口、模块发现和选择、插件库/商店/配置/修复。
- 外部二进制版本、架构、签名/哈希、来源和许可证记录。
- 注入前备份、错误隔离、超时、日志脱敏和禁用开关。

退出条件：主程序在模块缺失、损坏、架构不匹配或调用失败时仍可正常启动游戏；未经确认不发布任何来源不明二进制。

### Phase 7：更新、打包与兼容

交付：

- 本程序签名更新清单、下载校验、替换助手、回滚。
- `scripts/upstream-check.ps1` 与机器可读差异报告。
- 正式版本资源、便携包、干净机器验证和升级迁移测试。

退出条件：更新中断可回滚；压缩包内文件复核通过；所有 P0/P1 清单关闭。

## 7. 测试策略

- 单元测试：路径、配置、参数、版本比较、范围分类、状态机、清单和哈希。
- Win32 集成测试：消息循环、窗口销毁、托盘、DPI、COM、hook、热键、SendInput。
- 故障注入：网络断开、磁盘满、只读目录、锁文件、损坏 JSON、假清单、超时进程。
- 竞态检测：纯 Go 层运行 `go test -race`；Win32 层另做压力和句柄审计。
- 泄漏基线：goroutine、线程、HANDLE、USER、GDI 对象、私有工作集按场景前后对比。
- 打包复核：重新打开 ZIP，验证 EXE 版本、图标、配置模板、许可证、哈希和不应出现的开发文件。

## 8. 完成定义

每项功能只有同时满足以下条件才算完成：

1. 正常路径可用，并与上游基线的保留行为相符。
2. 取消、失败、重复操作、关闭和恢复路径已测试。
3. UI 线程无长任务，错误可见且日志可定位。
4. 相关稳定性条目有测试证据或明确“不适用”理由。
5. 文档、配置示例、打包内容和版本信息同步更新。
6. 对应上游来源提交和本项目实现状态可追踪。
