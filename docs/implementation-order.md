# 需求总表与唯一代码执行顺序

状态：正式执行基线  
制定日期：2026-07-17  
适用范围：本项目后续全部代码、测试、文档和发布工作

## 1. 本文档的作用

本文档把用户此前提出的需求整理为不可变约束、功能范围和实际代码阶段。后续工作以本文档的 `S00`～`S13` 为唯一顺序；[重构与实施路线](refactor-roadmap.md)负责解释架构，[功能范围矩阵](upstream-scope-matrix.md)负责解释上游功能取舍，[稳定性清单](go-win32-stability-checklist.md)负责逐项防止崩溃和卡死。

执行规则：

1. 同一时间只推进一个主阶段。
2. 前一阶段的退出条件未满足，不把下一阶段标为开始。
3. 可以提前做只读研究和接口预留，但不能提前合入未验收功能。
4. 每个阶段都必须同时交付实现、测试、文档和可运行验证结果。
5. 发现新的真实崩溃/卡死模式时，先补充稳定性清单，再修复和回归。
6. 上游发生变化时先生成差异和分类报告，不能打乱当前执行阶段。

当前进度：`S00`、`S01` 已完成；下一阶段为 `S02：稳定 Win32 外壳`。

## 2. 原始需求整理

### 2.1 项目与技术要求

1. 项目从空目录开始建立。
2. 使用 Go + Win32 实现 Windows x64 原生桌面程序。
3. 以 FufuLauncher/FufuLauncher 为行为参考，重构范围内的全部功能。
4. 不复刻上游花哨界面，采用简洁、时髦、低资源占用的暗色 UI。
5. 鼠标连点和键盘连按必须可靠，属于最高优先级功能。
6. 能自动对照上游更新，但不能自动带入被排除功能或未经审计的二进制。
7. 在实现过程中逐项对照 Go + Win32 崩溃、卡死和资源泄漏清单。
8. 开工前先完成重构规划、功能边界和路线文档；该项已经完成。

### 2.2 永久排除的功能

以下项目在任何阶段都不实现，也不保留入口、后台服务、数据库或网络请求：

- FufuLauncher 自身登录。
- 米游社、HoYoLAB、云游戏及社区登录。
- 账号保存、账号登录切换，以及游戏设置页内的账号切换。
- Cookie、Token、二维码、验证码、安全认证和刷新链路。
- 抽卡记录、抽卡统计和 UIGF。
- 签到、批量签到、云签到和便笺。
- 资讯、新闻、社区内容、公告内容浏览和兑换码内容。
- 原仓库控制面板/工具箱的全部功能。
- BBS、VanillaBBS 和相关 WebView 页面。
- 数据中心、成就、背包、玩家资料等账号数据页。
- 帮助文档、养成计算器和提瓦特地图内容页。
- “其他功能”中的附加程序随游戏启动。
- “实用工具”中的内置浏览器。
- 仅为以上功能服务的 WebView2、缓存、模型、数据库和服务。

### 2.3 需要实现的主要功能

- 简洁暗色 Win32 主窗口、导航、托盘、单实例、DPI、多语言和配置。
- 鼠标左键连点、鼠标右键连点、键盘连按、触发键、停止键、间隔和状态显示。
- 游戏路径自动发现、手动选择、版本/区服/目录状态识别和进程监控。
- 普通游戏启动、自定义参数、分辨率、显示器、窗口模式和启动后行为。
- 游戏资源下载、语音包、预下载、校验、修复、进度和取消。
- 官服/B服切换、高级服务器转换和失败回滚。
- HDR 设置、桌面快捷方式、启动声音和 BetterGI 联动。
- 游戏窗口截图、截图热键和保存目录。
- FPS/CPU/GPU 覆盖层。
- 注入开关、模块选择和纯净启动回退。
- 插件发现、启停、配置、预设、修复、更新和插件商店。
- 本程序诊断、版本信息、更新、回滚和便携包。
- 上游提交自动检查、范围分类和人工审计后更新基线。

## 3. 唯一代码执行顺序

### S00：需求冻结与上游基线

状态：已完成。

交付物：

- 重构路线、范围矩阵、稳定性清单和上游同步方案。
- `upstream.lock.json` 锁定参考仓库、分支、提交和范围策略。
- 本文档固定后续实施顺序。

退出条件：

- 所有原始需求在文档中有明确归属。
- 保留、排除和待审计项目互不矛盾。
- 当前阶段没有 Go 功能代码。

### S01：工程骨架与可重复构建

状态：已完成（2026-07-17）。

验收记录：[S01 verification record](s01-verification.md)。

目标：先让项目能稳定地构建、测试和识别版本，不开始业务功能。

代码工作：

- 初始化 Git 仓库和 Go module。
- 建立 `cmd/`、`internal/`、`scripts/`、`assets/`、`docs/`、`dist/` 基础结构。
- 固定 Go 版本、模块依赖策略、格式化、静态检查和测试命令。
- 建立项目本地 `GOTMPDIR`/`GOCACHE` fallback。
- 建立版本单一来源、Windows manifest、VERSIONINFO、图标和构建脚本。
- 建立 EXE 同级 `data/` 以及其中的 `logs/`、`cache/`、`staging/` 便携目录规范。
- 增加许可证、第三方来源和二进制哈希清单模板。

退出条件：

- Debug 和 GUI Release 均能一条命令构建。
- `go test ./...`、`go vet ./...` 和格式检查通过。
- 构建后的 EXE 能读取到正确文件版本和产品版本。
- 尚无实际游戏文件写入、网络请求、hook 或注入行为。

### S02：稳定 Win32 外壳

目标：建立后续所有功能共同依赖的、可诊断且能安全退出的 UI/线程基础。

代码工作：

- Win32 UI 主线程锁定 OS 线程并运行正确消息循环。
- 简洁暗色主窗口、左侧导航、状态栏和基础页面容器。
- Per-Monitor V2 DPI、窗口尺寸恢复、最小尺寸和可见区域校正。
- 单实例、第二实例唤醒、托盘显示/恢复/退出。
- UI dispatcher、后台任务 ID、取消机制和统一 shutdown coordinator。
- 结构化日志、顶层 panic 恢复、异常退出标记和诊断信息。
- 配置 schema、默认值、原子保存、损坏隔离和旧版本迁移框架。
- 句柄、线程、goroutine、USER/GDI 对象基础计数。

退出条件：

- 窗口和托盘重复打开/恢复/退出压力测试通过。
- 后台测试任务不会冻结 UI，退出时没有残留进程。
- 配置截断或损坏不会造成闪退。
- 稳定性清单 A、B、C、E、F、G 中与外壳有关的 P0 项关闭。

### S03：鼠标连点与键盘连按 P0

目标：在启动器复杂功能之前，把用户最关心的输入功能做成独立可靠引擎。

代码工作：

- 独立 hook OS 线程、`WH_KEYBOARD_LL`、`WH_MOUSE_LL` 和消息循环。
- Disabled/Armed/Running/Stopping/Fault 单一状态机。
- 键盘按住触发键时重复目标键的完整 down/up。
- 鼠标左键和右键按住连点。
- 独立停止键、间隔设置、安全下限和实时状态。
- injected flag 与自定义 `dwExtraInfo` 双重过滤，防止自激。
- 扫描码、扩展键、左右修饰键和键盘布局处理。
- `SendInput` 返回值、UIPI/完整性级别检测和明确错误。
- 禁用、松开、切模式、退出、锁屏、休眠时强制释放虚拟按键。
- 独立输入测试页/测试程序和延迟、频率、down/up 计数。

退出条件：

- 记事本键盘连按和原始输入鼠标连点各连续运行 10 分钟。
- 1000 次快速触发、200 次启停后只存在一个输出循环且资源回到基线。
- 任意停止路径都不会出现粘键、松开后继续输出或关闭后继续输出。
- 普通/管理员权限组合要么正常工作，要么显示准确权限错误。
- 稳定性清单 D 类全部 P0 项关闭。未通过时禁止进入 S04。

### S04：游戏发现、路径与只读状态

目标：先正确认识本机游戏状态，不立即修改或启动游戏。

代码工作：

- 自动发现可能的游戏目录和候选 EXE。
- 手动选择游戏目录。
- 读取 `config.ini`、游戏版本、渠道/区服和安装状态。
- 后台计算目录大小，支持取消和进度。
- 游戏进程检测、HANDLE/创建时间跟踪和运行状态显示。
- Unicode、空格、长路径、多候选、无权限和损坏配置处理。

退出条件：

- 对有效、无效、多个候选和自定义 EXE 名称均给出确定结果。
- 只读扫描不修改游戏目录和注册表。
- 大目录扫描期间 UI 可操作且可以立即取消。

### S05：纯净启动与启动设置

目标：先完成不依赖注入和插件的可靠游戏启动链路。

代码工作：

- 普通/纯净启动、Windows 参数 quoting 和显式工作目录。
- 自定义命令行参数。
- 分辨率预设、自定义宽高、目标显示器和窗口模式。
- 启动后保持、最小化到托盘或退出行为。
- 游戏启动、快速退出、已运行和启动失败状态机。
- 桌面快捷方式。
- 不在此阶段接入注入、插件或附加程序。

退出条件：

- 路径带空格/Unicode、自定义参数、多显示器和各窗口模式通过。
- 游戏快速退出或 UAC 取消时 UI 状态能恢复。
- 退出启动器不会误杀用户已有游戏进程。
- 纯净启动成为后续注入失败时始终可用的回退路径。

### S06：资源下载、预下载、校验与修复

目标：安全完成游戏资源管理，任何中断都不能破坏现有安装。

代码工作：

- 下载/预下载 manifest 和 schema 校验。
- 基础游戏、语音包、并发限制、进度、速度、剩余时间和取消。
- 磁盘空间预检、断点续传、超时和有界重试。
- 流式哈希、文件校验和修复计划预览。
- staging、事务日志、原子替换、备份、恢复和残留清理。
- API/schema 变化时失败关闭，不把零文件误报为完成。

退出条件：

- 断网、限速、取消、磁盘满、文件锁定、哈希错误和程序异常退出均测试。
- 正式游戏文件只在完整校验通过后替换。
- 取消或失败后原有可用游戏文件不受影响。

### S07：服务器、本地增强与外部联动

目标：完成剩余本地游戏管理功能，并保持每项修改可回滚。

代码工作：

- 官服/B服识别和切换。
- 高级服务器转换的变更预览、备份、事务和回滚。
- HDR 开关、HDR 亮度及注册表值备份恢复。
- 启动声音。
- BetterGI 联动；先审计路径、参数、进程生命周期和许可证。
- 明确保证这里的 BetterGI 联动不是已排除的任意“附加程序”配置入口。

退出条件：

- 区服切换中途失败可完整恢复原配置和 SDK 文件。
- 注册表类型错误或无权限不会破坏旧值。
- BetterGI 缺失或启动失败不影响纯净游戏启动。

### S08：截图与性能覆盖层

目标：实现与游戏窗口相关、但不依赖注入插件的辅助能力。

代码工作：

- 游戏窗口截图、组合热键、保存目录和文件命名。
- 截图 hook 与 S03 输入 hook 协调和冲突检测。
- FPS、CPU、GPU 采样和透明 click-through 覆盖层。
- 多显示器、DPI、无边框、全屏、最小化和游戏窗口重建处理。
- 游戏退出后停止捕获、计数器、ETW、计时器和覆盖层窗口。

退出条件：

- 连点/连按与截图同时启用时互不自触发、互不破坏卸载。
- 连续截图不会并发耗尽内存或阻塞 hook/UI。
- 覆盖层不拦截鼠标，游戏退出后资源回到基线。

### S09：注入适配层

目标：在纯净启动完全可靠后，隔离实现高风险注入能力。

代码工作：

- 审计上游注入模块的仓库、许可证、ABI、版本和二进制来源。
- 注入开关、模块选择、兼容矩阵和版本协商。
- 检查 PE 架构、导出、文件版本、SHA-256 和游戏版本。
- 优先通过独立 helper 进程隔离模块崩溃和超时。
- 注入失败回到明确错误状态，并允许立即改用纯净启动。
- 展示反作弊、游戏崩溃和账号风险，不实现规避或隐藏检测。

退出条件：

- 模块缺失、损坏、错架构、错版本、超时均不导致主程序崩溃或卡死。
- 未知游戏/模块版本组合默认阻止注入。
- 关闭注入后运行路径与 S05 纯净启动一致。

### S10：插件管理、配置与插件商店

目标：在注入适配稳定后实现插件生态，不让单个插件破坏主程序。

代码工作：

- 插件发现、启停、排序、重命名和安全删除。
- 插件配置 schema、编辑、预设、损坏回退和原子保存。
- 插件下载、安装、修复、更新、版本目录和回滚。
- 插件商店搜索、分类、排序、分页和已安装状态。
- 来源、许可证、SHA-256、兼容范围和更新端点记录。
- 安全模式：跳过全部插件仍能启动程序和纯净游戏。

退出条件：

- 插件路径不能利用 `..`、junction 或符号链接逃逸插件根目录。
- 商店/插件损坏、断网或更新中断不会阻止主程序启动。
- 插件不能引入登录、账号、BBS、资讯或其他永久排除模块。

### S11：程序设置、语言与 UI 收尾

目标：在功能结构稳定后完成设置页和视觉一致性，不提前为未完成页面反复返工。

代码工作：

- 简中和英文资源，保留后续语言扩展结构。
- 暗色主题、跟随系统、高对比度 fallback；不做复杂皮肤和视频背景。
- 托盘、窗口尺寸、最小尺寸、程序优先级和 CPU 异常提醒设置。
- 页面键盘导航、焦点、无障碍文本和错误状态。
- 关于页、版本、上游审计 SHA、许可证和诊断导出。
- 删除所有未实现占位入口和永久排除功能残留字符串。

退出条件：

- 100%/125%/150%/200% DPI 和多显示器布局可用。
- 所有保留功能有入口，所有排除功能没有入口或后台初始化。
- 空闲 CPU、内存、句柄、USER/GDI 对象符合轻量目标。

### S12：本程序更新与上游自动对照

目标：实现两套相互独立的更新能力：本程序安全更新，以及上游只读差异审计。

代码工作：

- 本程序 SemVer、签名更新清单、SHA-256、下载、替换 helper 和回滚。
- `scripts/upstream-check.ps1` 或等价 Go 工具。
- 读取 `upstream.lock.json`，比较 base/head 提交和变更文件。
- 生成 `in_scope`、`excluded`、`review_required`、`dependency_risk` 报告。
- 只有人工审计并关联实现/测试后才能更新上游基线。
- 上游二进制、API、schema、许可证和注入变更必须人工复核。

退出条件：

- 更新断网、校验失败、替换失败和断电模拟均可回滚。
- 上游 API 失败或限流不会改变 lock 文件。
- 自动对照不会运行上游脚本或二进制，也不会自动恢复永久排除功能。

### S13：全量回归、打包与首个正式版本

目标：把各阶段通过的功能作为整体重新验证并形成可复现发布物。

代码工作：

- 全量单元、集成、竞态、故障注入、压力、休眠/唤醒和权限测试。
- 长时间输入、下载、进程监控、截图、覆盖层、注入和插件组合测试。
- 正式 VERSIONINFO、manifest、图标、许可证和第三方通知。
- 生成便携目录和 ZIP，重新打开 ZIP 核对实际装载文件。
- 干净 Windows 10/11 x64 环境验证首次运行、升级和回滚。
- 完成发布说明、已知风险、上游审计基线和 SHA-256。

退出条件：

- 没有开放的 P0/P1 稳定性问题。
- 所有保留功能通过验收，所有永久排除功能经二进制/字符串/网络行为审计确认不存在。
- 源码、文档、便携目录、ZIP 和版本信息一致。
